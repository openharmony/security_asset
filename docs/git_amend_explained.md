# Git commit --amend 详解：为什么它会改变 Commit ID

## 1. 核心概念：Commit ID 是内容的指纹

Git 的 Commit ID（SHA-1 hash）不是分配的序号，而是由以下字段计算出来的**指纹**：

```
Commit Hash = SHA1(
    Tree Hash        (当前目录结构的快照)
  + Parent Hash      (父提交的 hash)
  + Author Name      (作者)
  + Author Date      (作者时间)
  + Committer Name   (提交者)
  + Committer Date   (提交时间)  ← amend 会更新这个！
  + Commit Message   (提交信息)
)
```

**任何一项发生变化，hash 就会改变。** 这和"复用旧 ID"是不兼容的 — 你不可能让两个不同的内容拥有同一个指纹。

> 类比：就像文件的 MD5 校验值，修改了哪怕一个字节，校验值就完全不同。

---

## 2. --amend 的本质：不是"修改"，而是"替换"

`git commit --amend` 的实际操作如下：

```
                    ┌──────────────┐
                    │  old commit  │  hash = b723bc0a
                    │  tree: A     │
                    │  parent: X   │
                    │  date: T1    │
                    │  msg: "foo"  │
                    └──────────────┘
                           │
           git commit --amend
                           │
                           ▼
                    ┌──────────────┐
                    │  new commit  │  hash = 8be78200  ← 全新 hash！
                    │  tree: B     │  ← tree 变了（你改了代码）
                    │  parent: X   │  ← parent 不变
                    │  date: T2    │  ← 时间变了！
                    │  msg: "bar"  │  ← message 可能变了
                    └──────────────┘
```

关键步骤：
1. Git 用 **新的内容**（新的 tree、新的时间等）计算出一个 **新的 hash**
2. 将分支指针（如 `pr_413`）从旧 hash 移到新 hash
3. 旧提交 `b723bc0a` 变成孤儿（无人引用），等待 GC 回收

**Git 从不修改已有提交对象，它只创建新对象。** 这是不可变数据模型。

## 3. 图示：amend 前后的分支状态

### 3.1 amend 前：本地和远程一致

```
  Remote (master)                        Local (pr_413)
  ──────────────                         ──────────────

       ┌──────────┐                          ┌──────────┐
       │ 817d8266 │                          │ 817d8266 │
       └──────────┘                          └──────────┘
             │                                     │
             ▼                                     ▼
       ┌──────────┐                          ┌──────────┐
       │ 0a371083 │                          │ 0a371083 │
       └──────────┘                          └──────────┘
             │                                     │
             ▼                                     ▼
       ┌──────────┐                          ┌──────────┐
  HEAD→│ b723bc0a │ ← ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ →│ b723bc0a │ ← HEAD
       │ "修正json"│         同一个 commit    │ "修正json"│
       └──────────┘                          └──────────┘

  ✅ 本地 tip == 远程 tip，可以快进推送 (fast-forward)
```

### 3.2 amend 后：本地产生了新 hash

```
  Remote (master)                        Local (pr_413)
  ──────────────                         ──────────────

       ┌──────────┐                          ┌──────────┐
       │ 817d8266 │                          │ 817d8266 │
       └──────────┘                          └──────────┘
             │                                     │
             ▼                                     ▼
       ┌──────────┐                          ┌──────────┐
       │ 0a371083 │                          │ 0a371083 │
       └──────────┘                          └──────────┘
             │                                     │
             ▼                                     │
       ┌──────────┐                          ┌──────────┐
  HEAD→│ b723bc0a │                          │ 8be78200 │ ← HEAD
       │ "修正json"│                          │ "修正json"│
       └──────────┘          ❌ 分叉！        └──────────┘
             │                不同的 hash         │
             │                                     │
             └─────────── 远程还停在旧提交 ────────┘

  ❌ 远程 tip 是 b723bc0a，本地 tip 是 8be78200
     → 历史 diverged，无法快进推送！
```

### 3.3 推送失败的原因

```
  git push my_remote pr_413:master

  远程检查逻辑：
  ┌──────────────────────────────────────────────────────┐
  │  远程 master tip = b723bc0a                          │
  │  本地 pr_413   tip = 8be78200                        │
  │                                                      │
  │  8be78200 的 parent 是 0a371083                      │
  │  b723bc0a 的 parent 也是 0a371083                    │
  │                                                      │
  │  两个提交从同一个 parent 分叉出去了！                  │
  │  这不是 fast-forward，远程拒绝！                      │
  └──────────────────────────────────────────────────────┘
```

## 4. 解决方案对比

### 方案 A：Force Push（危险）

```
  git push --force-with-lease my_remote pr_413:master

  效果：强制让远程指向 8be78200，丢弃 b723bc0a

  Remote (master) 修改后：
  ──────────────────────

       ┌──────────┐
       │ 817d8266 │
       └──────────┘
             │
             ▼
       ┌──────────┐
       │ 0a371083 │
       └──────────┘
             │
             ▼
       ┌──────────┐
  HEAD→│ 8be78200 │  ← 远程被强制改指向新 hash
       └──────────┘

       b723bc0a 变成孤儿 ⚠️ 如果其他人基于 b723bc0a 开发，
       他们的历史会断裂！
```

### 方案 B：Soft Reset + New Commit（安全）

```
  git reset --soft b723bc0a    # 回到远程一致的 commit，修改保留在 staging
  git commit -m "修正json解析逻辑"  # 新建一个 commit 接在 b723bc0a 后面

  执行过程：

  Step 1: reset --soft b723bc0a
  ──────────────────────────────

       ┌──────────┐
       │ 817d8266 │
       └──────────┘
             │
             ▼
       ┌──────────┐
       │ 0a371083 │
       └──────────┘
             │
             ▼
  HEAD→┌──────────┐
       │ b723bc0a │  ← HEAD 回到这里，和远程一致！
       └──────────┘
       [staging area: 8be78200 和 b723bc0a 的所有差异]

  Step 2: git commit -m "修正json解析逻辑"
  ───────────────────────────────────────────

       ┌──────────┐
       │ 817d8266 │
       └──────────┘
             │
             ▼
       ┌──────────┐
       │ 0a371083 │
       └──────────┘
             │
             ▼
       ┌──────────┐
       │ b723bc0a │  ← 远程 tip
       └──────────┘
             │
             ▼
  HEAD→┌──────────┐
       │ NEW HASH │  ← 新提交接在 b723bc0a 后面
       └──────────┘

  ✅ 这是 fast-forward！可以正常推送。
```

---

## 5. 总结

| 特性 | commit --amend | reset --soft + commit |
|------|---------------|----------------------|
| 结果 | 替换旧提交（新 hash） | 新增提交（旧 hash 保留） |
| 历史 | diverged（分叉） | fast-forward（快进） |
| 推送 | 需要 --force | 正常推送 |
| 安全性 | 可能覆盖他人工作 | 安全 |

### 核心原则

- **已推送的提交 = 公共历史，不要 amend**
- **未推送的提交 = 私有历史，可以放心 amend**
- `--amend` 本质是"删除旧提交 + 创建新提交"，不是"原地修改"
- Commit ID 是内容的 SHA-1 指纹，内容变了 ID 必然变，这是数学规律而非 Git 设计选择
